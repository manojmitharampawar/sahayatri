01 · The problem

Why Indian train travel needs Sahayatri

India's railways carry over 8 billion passengers a year across the world's largest rail network. Yet managing a train journey — from booking confirmation to safe arrival — is fragmented across half a dozen disconnected apps, none of which work reliably offline or talk to each other.

A typical passenger on a long-distance journey today faces all of these at once:

  • Manually checking PNR status across NTES and third-party apps before every journey
  • Guessing when to leave home, with no dynamic adjustment for train delays
  • No reliable way to track physical position on the route during the journey
  • Anxiety about missing their destination station — especially on overnight journeys in Vishram (sleep)
  • Complete helplessness when cellular connectivity drops in rural and mountainous stretches
  • No single visual showing where their train actually is on an Indian railway map

No existing application solves all of these together. Sahayatri does.

02 · Product vision

What Sahayatri is, and what it is not

Sahayatri is a mobile travel companion that eliminates the mental load of the Indian train journey — from automatic booking detection to a GPS-triggered arrival alarm — using intelligent parsing, real-time railway data, offline GPS tracking, and an interactive Indian Railways track map built from actual physical shapefile data.

Sahayatri's foundational promise: once a journey is loaded, the app works with zero data connectivity for the entire duration. The server is consulted when available; it is never required.

The product launches with Indian Railways (Phases 1–4) and is architected from the start to extend to flights, buses, and metro (Phase 5). Trains are the ideal foundation — they involve the longest journeys, the most acute need for offline functionality, the most structured booking data, and the largest addressable user base in India.

03 · Core product pillars

Four features. One companion.
Yatra Khoj यात्रा खोज Journey detection

Sahayatri automatically scans the user's email inbox (Gmail, Outlook via OAuth) and SMS messages for IRCTC booking confirmations and PNR-linked messages. Every relevant detail is extracted and populated into a clean Yatra Card — without the user entering a single value manually.

  • PNR number, train name and number, class of travel
  • Boarding and destination station name, code, and geo-coordinates
  • Scheduled departure and arrival times
  • Coach, seat or berth number, passenger names
  • All parsing is on-device — raw message content is never transmitted to a server
Samay Suchna समय सूचना Time intelligence

Before the journey, Sahayatri watches two live signals simultaneously: the train's running status (via NTES / Where Is My Train API) and the user's GPS position relative to the boarding station. These are combined into a single actionable answer to the passenger's real question: when do I need to leave?

  • Nikalne ka waqt — dynamic "leave now" alert adjusted for both live delay and current commute time to station
  • If the train is 45 minutes late, the departure alert is pushed forward 45 minutes automatically
  • Progressive delay escalation — as delay grows, nudge frequency and urgency increase
  • Live on-time / late indicator on the Yatra Card with estimated platform arrival time
Safar Rakshak सफर रक्षक Journey guardian

Once the journey begins, Sahayatri becomes a guardian. It switches to GPS-based position tracking, mapping real-world coordinates against the pre-loaded Indian Railways track shapefile. This works entirely offline — GPS hardware operates via satellite, independent of cellular connectivity.

  • Continuous Haversine distance calculation from current position to destination station
  • GPS coordinates snapped to nearest point on track shapefile — train icon always on the line, never floating over fields
  • Breadcrumb trail stored locally in SQLite — full positional history of the journey
  • Variable GPS polling rate: 15 min / 5 min / 60 sec based on distance threshold
  • Batch sync to server when connectivity returns — never blocks the offline experience
  • Utarne ki alarm — geofence-triggered destination alarm based on physical distance (GPS), not scheduled time
  • Vishram Mode — all notifications silenced except the Utarne ki alarm; passengers sleep peacefully on overnight journeys
The smart alarm is the defining feature. A traditional alarm wakes the passenger at 4:00 AM. Sahayatri checks GPS: the train is still 100 km away. The alarm is silently suppressed. At 5:50 AM, the geofence fires — the user wakes up 10 minutes from their station, regardless of delay.
Rail Darshan रेल दर्शन Track visualiser

Sahayatri ships with a full station-by-station map of Indian Railways routes built from physical track geodata (OpenStreetMap + OGD India). The user's train appears as a live position dot moving along the actual track — not a road map.

  • Physical railway shapefiles rendered as a vector map layer (track-only view, uncluttered)
  • All intermediate stations labelled with name, code, and distance from current position
  • Live train position updated from GPS breadcrumb data and snapped to the shapefile
  • Expected time of arrival at each upcoming station
  • Vilamba flag — delayed train's position dot visually lags behind its scheduled position on the map, making delay intuitively obvious without reading numbers
  • Map works fully offline after a one-time tile pack download (Mapbox offline caching)
  • Shareable live view — family at the destination station can watch the dot and know exactly when to head to the platform

03.5 · First journey flow

Onboarding, permissions, and the empty state

Progressive permission request

Asking for SMS, Location, and Email OAuth permissions all at once causes massive drop-off. Sahayatri uses a stepped permission flow that builds trust incrementally:

  • Step 1 — Location (at install): Required for core GPS tracking. Explained as "so we can wake you up before your station"
  • Step 2 — Email OAuth (first launch): Presented with a clear value proposition: "We'll find your IRCTC bookings automatically." Offer "Skip — I'll enter my PNR" as an equal-weight alternative
  • Step 3 — SMS (deferred, Android only): Requested only after the user has completed one journey or explicitly taps "Scan SMS for bookings." Never requested on first launch

Empty state design

A user who installs Sahayatri with no upcoming journey must not see a blank screen. The empty state shows:

  • A prominent "Enter PNR" field as the primary call-to-action
  • A secondary "Scan QR code from e-ticket" button
  • A brief animated illustration showing how the Yatra Card works
  • A "Connect Email" card if OAuth has not been granted yet

Transparency receipt

After every email or SMS scan, Sahayatri displays a transparency receipt:

"We scanned your inbox and found 1 IRCTC booking. Here's what we extracted: [PNR, train name, date]. No other messages were read or stored."

This builds trust with privacy-conscious users and satisfies Google Play's permission rationale requirements for READ_SMS.

03.6 · Shared journeys

Group travel, family tracking, and co-passenger features

Yatra Card sharing

Indian rail travel is inherently social — families book 4–6 tickets together. One person installs Sahayatri; others need the same alerts. The Yatra Card supports sharing via:

  • Deep link: A unique URL that opens the Yatra Card in the recipient's Sahayatri app (or prompts install)
  • QR code: Displayed on-screen for in-person sharing at the station
  • WhatsApp / SMS share: One-tap share button generates a pre-formatted message with the deep link

Family live view

The "shareable live view" from Rail Darshan is a lightweight web page — no app install required. Family at the destination station open the link in any browser and see the train dot moving on the map in real time.

  • Link is generated from within the Yatra Card with a single tap
  • Web view shows: train position on map, current speed, ETA at destination, delay status
  • Link expires automatically 1 hour after arrival at the destination station
  • No login required for viewers — read-only, anonymous access

Co-passenger awareness

After chart preparation, IRCTC reveals nearby passengers. Sahayatri can surface this as a safety and social feature:

  • Display coach and berth layout with occupied/vacant indicators
  • Optional: allow Sahayatri users in the same coach to see each other (opt-in only)

03.7 · Post-arrival experience

The journey doesn't end at the station

Journey summary card

After arrival, Sahayatri generates a visual journey summary:

  • Total distance travelled and duration
  • Number of stations passed
  • Delay encountered (total minutes, max delay point)
  • Route map with the breadcrumb trail overlaid on the track shapefile
  • Shareable as an image card (Instagram story / WhatsApp status format)

Alarm calibration feedback

A single-question feedback prompt after each journey:

"Did the alarm wake you at the right time?" — Too early / Just right / Too late

This feedback is used to calibrate the geofence threshold per user. If a user consistently reports "too early," the default 5 km radius is tightened. If "too late," it is expanded. Over time, the alarm becomes personalised.

Journey history

All past journeys are archived in a timeline view, useful for:

  • Frequent travellers tracking their routes
  • Expense reports and reimbursement documentation
  • Nostalgia — revisiting past journeys on the map

Contextual app review prompt

The app store review prompt is shown only after a successful alarm wake-up — the moment of highest user satisfaction. Never shown on first journey or after a failed alarm.

04 · Technical specification

The algorithms and logic that make Sahayatri work

4.1 Distance engine — Haversine formula

All distance calculations between the device's GPS position and any station coordinate use the Haversine great-circle formula. This accounts for the Earth's spherical curvature and delivers accurate results over India's vast rail network, which spans thousands of kilometres across varying terrain:

d = 2r · arcsin( √( sin²((φ₂ − φ₁) / 2) + cos(φ₁) · cos(φ₂) · sin²((λ₂ − λ₁) / 2) ) ) φ = latitude in radians | λ = longitude in radians | r = 6,371 km (Earth's mean radius)

This runs entirely on-device in native Kotlin (Android) or Swift (iOS) — no external library, no network request. The resulting distance d is recalculated at every GPS poll and compared against the user's alarm threshold to determine whether to fire the Utarne ki alarm or escalate to a higher polling tier.

4.2 GPS-to-track snapping

Raw GPS coordinates have accuracy of ±10–50 metres and can place the train icon off the track, especially in the Western Ghats, the Northeast hill regions, or urban corridors where multipath signal interference is common. Sahayatri solves this in two steps.

First, the app bundles Indian Railways shapefiles — geographic path data describing the physical route of every rail line in India — as a compressed offline asset. A spatial index (R-tree or KD-tree) is built over this data at first launch for fast nearest-point lookup. When a new GPS coordinate arrives, the app finds the nearest point on the shapefile and snaps the position marker to it. The train icon always appears on the track.

Second, for stretches with parallel tracks at major junctions, the snapping algorithm uses the known route (loaded from the Yatra Card) to select the correct track segment — preventing the icon from jumping to an adjacent line.

Kalman filter pre-processing

Raw GPS coordinates jump erratically, especially at low speeds or in urban canyons. Before snapping, Sahayatri applies a lightweight Kalman filter to smooth the position stream. This reduces visual jitter on the Rail Darshan map and improves snapping accuracy by filtering out transient noise spikes.

Speed-weighted snapping confidence

A GPS point 200 metres off-track at 80 km/h is noise — the same point at 0 km/h might mean the train has stopped at a siding or yard. The snapping algorithm uses instantaneous speed as a confidence weight: at high speed, aggressive snapping to the nearest track point; at low speed or stationary, the raw position is preserved with a wider tolerance band.

Tunnel and GPS-gap interpolation

GPS signal is lost in tunnels and deep cuttings. When no GPS fix is received for more than 30 seconds, Sahayatri switches to dead-reckoning interpolation: the map dot continues moving along the track shapefile at the last known speed. When GPS signal returns, the position is smoothly corrected to the actual fix. This prevents the dot from "freezing" and then "jumping" — a common and disorienting UX failure in competing apps.

4.3 Variable polling rate and battery management

Running GPS at full frequency on a 24-hour overnight journey drains a phone battery long before arrival. Sahayatri implements a three-tier adaptive polling strategy:

Variable polling rate zones — GPS frequency adapts automatically as the train approaches the destination. Average GPS-active time stays under 5% of total journey duration during the cruise phase.

4.3.1 API fallback waterfall for Samay Suchna

Samay Suchna depends on external APIs for live train status. A single-source dependency is unacceptable for a travel-critical app. Sahayatri implements an explicit fallback waterfall:

NTES API (primary, 30s timeout) → RailYatri API (secondary, 20s timeout) → Where Is My Train API (tertiary, 15s timeout) → Last known status + "stale data" badge on Yatra Card

Polling cadence adapts to journey proximity:

Time to departureStatus poll interval
More than 6 hoursEvery 15 minutes
2 – 6 hoursEvery 5 minutes
Under 2 hoursEvery 2 minutes
Under 30 minutesEvery 1 minute

If all three API sources fail simultaneously, the Yatra Card displays the last known delay value with a timestamp and a "Data may be outdated" warning. The GPS-based tracking and alarm system continues to function independently — it never depends on API availability.

4.4 Smart GPS-adjusted alarm — not clock-adjusted

This is the defining feature of Safar Rakshak. Conventional travel alarms fire at a scheduled time. If the train is 2 hours late, the alarm fires when the train is still 150 km from the destination — waking the passenger unnecessarily and leaving them stranded in the early hours.

Sahayatri's Utarne ki alarm is driven by GPS distance, not the clock. The geofence fires when the Haversine distance d falls below the user's threshold (default 5 km, configurable from 2 to 20 km). The same logic governs the pre-journey departure alert — it is held or released based on live delay and current GPS position from home, not a fixed clock offset.

4.5 Breadcrumb trail and offline continuity

GPS hardware operates via satellite signal independent of cellular networks — it functions in the Chambal ravines, the Western Ghats, and every tunnel in between. Sahayatri records every GPS coordinate as a breadcrumb point in a local SQLite database, enabling:

  • Smooth track map position interpolation during cellular signal gaps
  • Speed-based ETA recalculation using average speed over recent breadcrumbs
  • Full journey trajectory synced to the server as a single batch when connectivity returns
  • Post-journey route review for the user
The offline guarantee

Everything required for the journey — the track shapefile, station coordinates, timetable, GPS alarm logic, and offline map tiles — is resident on-device before departure. Sahayatri never requires a network connection to protect the passenger during travel.

4.6 Failure modes and graceful degradation

The current design is optimised for the happy path. Real-world Indian rail travel involves frequent edge cases that must be handled gracefully:

ScenarioDetection methodGraceful degradation
Train cancelled NTES API status check; IRCTC cancellation SMS/email Yatra Card shows "Cancelled" badge; all alarms disabled; refund status link displayed
Waitlisted ticket (WL → RAC → CNF) Periodic PNR status polling via NTES API Yatra Card shows live WL position with trend indicator (moving up / stuck); visual confirmation when status changes to RAC or CNF; "Not confirmed" warning badge until chart preparation
Platform change at station NTES platform assignment API (where available) Push notification with updated platform number; Yatra Card updated in real time; fallback: "Platform not yet assigned" with manual check link
GPS hardware failure mid-journey No GPS fix received for 3 consecutive polling cycles Fall back to last-known-speed dead reckoning; show "GPS signal lost" indicator on map; continue alarm logic using estimated position; re-acquire GPS automatically when signal returns
Multiple journeys on same day Multiple Yatra Cards with overlapping or sequential dates Journey timeline view showing all legs; automatic transition from completed journey to next leg; independent alarm and tracking for each segment
Chart not yet prepared PNR status API returns "Chart Not Prepared" Yatra Card shows "Chart pending" badge; coach/berth fields show "Will update after chart preparation"; automatic refresh when chart is published (typically 4 hours before departure)

07 · Platform considerations

Android and iOS behave differently — both paths are designed

Android

READ_SMS permission is available but scrutinised by Google Play — typically approved only for messaging-primary apps. Sahayatri must accompany the request with a clear permission rationale screen. All SMS reading is on-device only; filter strictly to known IRCTC sender IDs. Offer manual PNR entry and QR scan as alternatives for users who decline. The journey experience (tracking, alarms, map) is unaffected by this choice.

iOS

Third-party apps cannot read SMS. The primary detection flow relies on email parsing via Gmail / Outlook OAuth. Secondary options: a Share Sheet extension (user shares booking message into Sahayatri from any app), manual PNR entry with auto-fill, or camera-based QR code reading from e-tickets. The full journey experience — GPS tracking, Vishram Mode, Rail Darshan, Utarne ki alarm — is identical on both platforms once the journey is loaded.

Privacy principles

  • All message parsing (email and SMS) happens on-device — no raw content is sent to any server
  • Only extracted structured data (PNR, stations, times) is stored — original messages are never retained by the app
  • GPS breadcrumbs are stored locally and synced only in aggregate, never streamed in real time
  • User can delete all journey data and location history from within the app at any time
  • No advertising, no third-party analytics SDKs with access to journey or location data

08 · Phased roadmap

Train first — multimodal later

Indian Railways is the right foundation: it is the most complex transport mode (longest journeys, most acute offline need, most structured booking data) and the largest addressable base. Every data model and infrastructure decision made in Phase 1–4 is designed to carry forward to flights, buses, and metro in Phase 5.

PhaseMilestoneScope
Phase 1 Yatra Khoj Email / SMS parsing, Yatra Card display, PNR status check, manual PNR entry fallback
Phase 2 Samay Suchna Live train status integration, Nikalne ka waqt alert, delay escalation, geo-adjusted departure nudge
Phase 3 Safar Rakshak Offline GPS tracking, variable polling (15 min / 5 min / 60 sec), Utarne ki alarm, Vishram Mode, breadcrumb sync
Phase 4 Rail Darshan Track shapefile rendering, GPS-to-track snapping, live position marker, station ETAs, Vilamba delay flag
Phase 5 Multi-modal Flight journey parsing (e-ticket / PNR), bus tracking, metro support, unified Yatra timeline across all modes

09 · Notification design system

Priority hierarchy, channels, and platform-specific behaviour

Sahayatri generates multiple notification types across the journey lifecycle. Without a clear priority hierarchy, notifications become noise — violating the core design philosophy. Every notification is classified into one of four priority tiers:

PriorityTypeBehaviourVishram Mode
P0 · Critical Utarne ki alarm Full-screen alarm with sound, vibration, and haptic. Overrides silent mode and DND. FIRES — the only notification that breaks Vishram
P1 · High Nikalne ka waqt, Train cancelled Heads-up notification with sound. Persistent until dismissed. Suppressed
P2 · Medium Delay escalation, Platform change, Chart prepared Standard notification. Grouped if multiple arrive within 5 minutes. Suppressed
P3 · Low Journey summary, Co-passenger info, Status sync Silent notification. Badge only, no sound or vibration. Suppressed

Android notification channels

Android 8+ requires notification channels. Sahayatri registers four channels matching the priority tiers, allowing users to independently mute delay nudges while keeping the arrival alarm at full volume:

  • sahayatri_critical — Utarne ki alarm only. Importance: MAX. Cannot be silenced by user (enforced by Android system for alarm-category notifications)
  • sahayatri_high — Departure alerts, cancellations. Importance: HIGH
  • sahayatri_medium — Delay updates, platform changes. Importance: DEFAULT
  • sahayatri_low — Informational. Importance: LOW

iOS Critical Alert entitlement

Standard iOS notifications can be silenced by Focus modes and DND — which passengers typically enable while sleeping. The Utarne ki alarm must bypass these. Sahayatri applies for Apple's Critical Alert entitlement (requires justification to Apple review team). This allows the alarm to play sound at system volume even when the device is in DND or Silent mode. The entitlement is used exclusively for the P0 Utarne ki alarm — never for any other notification.

10 · Security and OAuth token management

Protecting user credentials and journey data

OAuth token storage

  • Android: OAuth refresh tokens stored in Android Keystore (hardware-backed where available). Never in SharedPreferences or plain SQLite
  • iOS: OAuth refresh tokens stored in iOS Keychain with kSecAttrAccessibleAfterFirstUnlock protection class
  • Scope: Gmail API requests use gmail.readonly scope only. Outlook uses Mail.Read. No write or send permissions are ever requested

Token lifecycle

EventBehaviour
App launchToken health check — verify refresh token is valid; if expired, prompt re-authentication
Token revoked by user (via Google/Microsoft account settings)Graceful fallback to manual PNR entry; "Email sync paused" badge on home screen; no crash or error state
Token rotationRefresh tokens are rotated on every use (where supported by the OAuth provider); old tokens are immediately invalidated
App uninstallKeystore/Keychain entries are automatically purged by the OS

Data at rest

  • SQLite database encrypted with SQLCipher (AES-256) on both platforms
  • GPS breadcrumb data is stored locally and synced to server only in aggregate batches — never streamed
  • Shareable live view links use time-limited tokens (expire 1 hour after arrival) and contain no personally identifiable information

11 · Design philosophy

What Sahayatri feels like to use

The Indian train journey is a deeply human experience. People share berths, eat together, talk to strangers, watch the landscape pass in the dark. Sahayatri is designed to disappear into the background of that experience — not demand attention, not bombard with notifications, not turn the journey into a data dashboard.

It speaks when spoken to. It alarms when necessary. When the train is on time and the passenger is asleep, it does nothing at all — except watch.

Every product decision flows from this. Variable polling exists so the battery outlasts the journey. Vishram Mode exists so the passenger can close their eyes. The GPS-adjusted alarm exists so a late train does not steal sleep from a tired traveller. Rail Darshan exists so family at the destination can stop refreshing their phones and simply watch the dot move.

The name is not a feature name or a marketing term. It is a commitment. Sahayatri is your fellow traveller, and a fellow traveller does not let you miss your station.
